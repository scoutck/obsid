# v1.1 Design: AI-First Organization

Obsid v1.1 shifts from manual organization to AI-driven structure. The user writes; the AI tags, links, extracts people, and normalizes taxonomy in the background. Manual organization is available but never required.

## Guiding Principle

The user should never need to stop writing to organize. Organization happens invisibly on note close, with results visible as inline `#tags`, `[[links]]`, and person notes.

---

## 1. Data Model

### Changes to Note

- Add `unresolvedPeople String @default("[]")` — JSON array of names the AI couldn't confidently match to a known person.
- `tags` field remains as a **search cache** derived from inline `#tag` content. Content is the source of truth for tags.
- `type: "person"` identifies person notes. Title is the canonical name.

### New: PersonMeta

```
PersonMeta {
  id        String   @id @default(uuid())
  noteId    String   @unique
  aliases   String   @default("[]")  // JSON: ["Sarah", "Sarah C."]
  role      String   @default("")    // "Engineering Manager"
}
```

One-to-one with a Note (the person's note). Holds structured data the AI uses for name resolution. The note itself holds freeform content — observations, working style, context.

### New: NotePerson

```
NotePerson {
  noteId       String   // the note that mentions this person
  personNoteId String   // the person's Note ID (the note with type: "person")
  @@id([noteId, personNoteId])
}
```

Tracks which notes mention which people. Both fields are Note IDs — `noteId` is the mentioning note, `personNoteId` is the person's note. Enables queries like "all notes mentioning Sarah Chen."

### No changes to Collection

Manual collections are not extended in v1.1. Natural language queries via Ask Claude replace the need for new collection features.

## 2. `#tag` Inline Syntax & Autocomplete

### Source of truth

Tags live in the note body as `#tag` text. The `tags` DB field is a derived cache populated on save for fast search queries. Content is authoritative.

### CodeMirror extension (`src/editor/tag-syntax.ts`)

- Regex-based decoration matching `#word` patterns (alphanumeric, hyphens, underscores).
- Styled with `cm-tag` CSS class — visually distinct from body text.
- Ignores `#` in code blocks/inline code, headings (`# Heading`), and mid-word (`C#`).

### Autocomplete

Typing `#` triggers a filtered dropdown of existing vault tags, positioned at the cursor. Same keyboard navigation and positioning pattern as the slash menu. Tag vocabulary fetched from `GET /api/tags` (returns all unique tags with usage counts).

User can pick from the dropdown or type a new tag freely.

### Extraction on save

`extractInlineTags()` scans content for `#tag` patterns, returns a deduplicated array. Written to the `tags` DB field as a search index cache.

### AI-appended tags

When the AI organizes a note, it appends new tags as `#tag` text at the bottom of the note content. Only tags not already present in the content are added. The AI also normalizes near-duplicate tags (e.g., replaces `#meetings` with `#meeting-notes` if the latter is the established high-frequency variant).

## 3. Auto-Organize System

### Triggers

1. **Note close** — when the user navigates away from a note, a fire-and-forget `POST /api/ai/organize` fires in the background. Non-blocking.
2. **`/organize` slash command** — manual trigger on the current note. Same endpoint, but editor reflects content changes immediately since the user is still viewing the note. Toast notification for status.

### Request

The frontend sends `{ noteId, title, content, recentSiblings }` where `recentSiblings` is the last 5 edited note IDs (tracked client-side).

### Vault context (gathered server-side)

- Top 50 tags by frequency (from tag cache).
- Up to 100 recent note titles + first 100 characters.
- Content of the last 5 edited notes (recent siblings, sent as IDs and fetched server-side).
- Full people list: all PersonMeta records with name, aliases, role.

### AI call

Single prompt to **Claude Sonnet**. System instructions direct the AI to:

- Return structured JSON: `{ tags: [], links: [], people: [], unresolvedPeople: [] }`.
- Reuse existing tags. Normalize near-duplicates to established high-frequency versions.
- Only return new tags not already present in the note content.
- Match people against known aliases (case-insensitive). Use first-mention convention (e.g., "Sarah C." disambiguates subsequent bare "Sarah" within the same note).
- Skip ambiguous names — place in `unresolvedPeople`.
- Only suggest links to notes that actually exist in the vault.

### Response processing

1. **Tags** — filter out any already in content, append net-new as `#tag` at bottom of note.
2. **Links** — append net-new as `[[Link]]` at bottom of note (below tags).
3. **Tag normalization** — if AI flags a typo/variant tag in content, replace it with the established version.
4. **People (resolved)** — create NotePerson joins. Auto-create person notes (`type: "person"`) for newly encountered people, with PersonMeta stub (name as first alias, empty role).
5. **People (unresolved)** — store in `unresolvedPeople` field on the note.
6. Save updated content + unresolvedPeople to DB.

## 4. Person System

### Person notes

A person is a Note with `type: "person"`. Title = canonical name ("Sarah Chen"). Content = freeform markdown (observations, working style, preferences). Edited like any other note.

### AI auto-creation

When the organize step encounters a new, unambiguous person, it creates:
- A Note with `type: "person"`, title = extracted name, content = initial observation from the triggering note.
- A PersonMeta with the name as the first alias, role populated if detectable from context.

After creation, the AI does not modify person note content again. The person note becomes the user's to edit. The AI continues to read person notes for context during organize.

### Name resolution

1. AI extracts people mentions from note content.
2. Checks each name against PersonMeta aliases (case-insensitive).
3. **Match found** — create NotePerson join.
4. **Ambiguous** (e.g., bare "Sarah" with multiple Sarahs, no first-mention disambiguation) — add to `unresolvedPeople`.
5. **New person** — auto-create person note + PersonMeta.

### First-mention convention

The user writes last initial on first mention (e.g., "Sarah C."). The AI uses this to resolve subsequent bare "Sarah" within the same note. Aliases accumulate naturally: PersonMeta for "Sarah Chen" gains aliases `["Sarah Chen", "Sarah C.", "Sarah"]`.

### Adding to person notes without navigating

Use `/ask` (Ask Claude): "save this about Sarah" or "note that John prefers email over Slack." Claude finds the person note via `list_people` tool and appends using `update_note` with append mode. No new command needed.

Use `/claude` inline command (see Section 5): `/claude save this about Sarah` — same effect, fire-and-forget with confirmation line.

### `/people` slash command

- Lists all people with note count.
- Shows unresolved mentions across the vault: "Sarah (3 notes)" — pick which person to resolve.
- Merge duplicates: select two people, combine into one, merge aliases.

## 5. `/claude` Inline Command

### Purpose

General-purpose inline AI instruction. Type `/claude` followed by a natural language instruction anywhere in a note. Claude processes it in the background based on note context.

### Examples

- `/claude save this about Sarah` — appends observation to Sarah's person note.
- `/claude flag this risk` — could tag the note `#risk`.
- `/claude this should link to the Q2 plan` — inserts `[[Q2 Planning]]`.
- `/claude summarize the above into bullet points` — rewrites a section.

### Flow

1. User types `/claude save this about Sarah` inline in the note.
2. The slash menu appears at `/` as usual, showing "Claude Command" as an option. But the user keeps typing past the menu — once the input matches `/claude ` (with a space), the menu auto-dismisses and the editor enters "claude command mode." The user finishes their instruction and presses Enter.
3. On Enter, the full `/claude ...` line stays in the content. The editor fires a fire-and-forget `POST /api/ai/command` with `{ instruction, noteId, noteContent, noteTitle, cursorPosition }`.
4. Claude receives the note content, cursor context, and instruction. Uses existing vault tools to execute.
5. Confirmation line appended below the `/claude` line: `✓ saved to Sarah Chen's note`. On failure: `✗ couldn't resolve "Sarah" — multiple matches`.
6. Both the `/claude` line and confirmation are styled dimmed/subtle in the editor via the markdown preview extension.

### Cursor position

Sent with the request so Claude understands what "this" refers to. `/claude flag this risk` on line 12 means the content around line 12, not the whole note.

### Content integrity

- Confirmation lines are inserted only when the user is not actively editing that region (same debounce logic as auto-save).
- The AI organize step strips `/claude` and `✓`/`✗` lines before analyzing content, so they don't influence tagging or linking.
- The markdown preview extension renders these lines dimmed — present but visually quiet.

## 6. Enhanced Ask Claude

### New tools

- `list_people` — returns all people with name, aliases, role, and note count. Enables Claude to find the right person note for updates or queries.

### Updated tools

- `search_notes` — extended to filter by tags, people (via NotePerson join), and note type. Supports queries like "notes mentioning Sarah Chen" or "all person notes."

### Unchanged tools

- `read_note` — works for person notes since they're regular notes.
- `update_note` — works for appending to person notes (for `/claude save this about...` and conversational `/ask` requests).
- `create_note` — unchanged.

### Prompt updates

System prompt updated to inform Claude about:
- Person notes and their structure.
- The NotePerson relationship (which notes mention which people).
- The tag system (inline `#tags`, tag vocabulary).
- How to handle requests like "save this about [person]" — find person via `list_people`, append via `update_note`.

### No UI changes

`/ask` works exactly as before: inline prompt, response block, Keep/Dismiss. It simply has richer metadata to work with.

## 8. Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Organize trigger | On note close + manual `/organize` | Zero friction while writing. One API call per editing session. |
| AI approach | Single smart prompt (Approach A) | Holistic view of note + vault context in one reasoning pass. Highest quality. |
| Model | Sonnet for organize, Opus for Ask Claude | Organize is structured extraction (Sonnet excels). Ask Claude benefits from deeper reasoning. |
| Tag source of truth | Note content (inline `#tags`) | AI tags visible, editable, no hidden metadata. DB field is search cache. |
| Person model | Note with `type: "person"` + PersonMeta | Persons are notes — get editing, search, linking, AI organize for free. |
| Collections | No changes; Ask Claude replaces need | Natural language queries via enhanced Ask Claude. No manual filter UI needed. |
| Name disambiguation | First-mention convention + alias matching | User writes "Sarah C." first, AI resolves. Ambiguous names skipped, surfaced in `/people`. |
| `/claude` command text | Kept in note with confirmation appended | Full record of instructions and results. Both styled dimmed. |

## 9. Token Budget (Organize Prompt)

For a vault with ~500 notes and ~30 active tags:

| Component | Estimated tokens |
|-----------|-----------------|
| System prompt + instructions | ~500 |
| Note content | ~500-2000 |
| Top 50 tags with counts | ~200 |
| 100 note titles + snippets | ~2000 |
| 5 recent sibling summaries | ~1000 |
| People list (names + aliases) | ~300 |
| **Total input** | **~4500-6000** |
| **Output (JSON)** | **~200-500** |

Well within Sonnet's context window. Cost: ~$0.02 per organize call.
