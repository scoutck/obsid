# Obsid v1.2 Design: People-First Knowledge Base + Semantic Search

## Vision

Obsid v1.2 reorients the app around people as the primary axis of organization and adds semantic search via embeddings. The AI stops managing tags and instead focuses on building a living map of the user's professional relationships. The user never needs to organize — they just write, and the system builds the map.

### Guiding Principles

- **People are the connective tissue.** Most notes exist because of an interaction with a person. The app reflects that.
- **Capture should cost nothing.** If it takes more than 15 seconds to get an observation into the system, it won't happen.
- **The system gives back more than it takes.** Relationship context and semantic retrieval make it worth coming back to.
- **Search by meaning, not by label.** Embeddings replace tags as the retrieval mechanism.

## Scope

### In v1.2

- Embeddings on every note save — semantic retrieval replaces keyword search
- Tags become user-only — AI organize stops appending tags, autocomplete stays
- Chat mode — persistent conversation with Claude for observations, questions, actions
- People system overhaul — `/newperson`, `/pendingpeople`, enhanced person pages
- AI organize simplified — focuses on people links and wiki-links only

### Deferred

- Daily briefs / meeting prep
- Conversation browser (browsing old chat sessions)

## 2. Embedding Pipeline

### Trigger

Every note save, as part of the existing 500ms debounced auto-save. Chat-triggered note creates/updates also fire it.

### Flow

1. Note saved via `/api/notes/[id]` PUT
2. After DB write, fire-and-forget call to embed the note
3. Concatenate `title + "\n" + content` and send to Voyage API (`voyage-3`, 1024 dimensions)
4. Upsert into Embedding table as serialized Float32Array blob
5. If Voyage API fails, log and skip — note saves regardless, embedding retries on next save

### Semantic Search Function

New module: `src/lib/embeddings.ts`

1. Embed the query string via Voyage API
2. Load all embedding vectors from DB
3. Compute cosine similarity in JS (brute-force exact nearest neighbor)
4. Return top-N notes sorted by similarity score

Brute-force cosine over 1024-dim vectors is fast enough for personal knowledge base scale (sub-50ms for thousands of notes). Exact search — no approximation quality loss.

### Search Integration

- **Notes mode `/search`**: Embed the query, return semantically ranked results. Fall back to FTS5 if embedding API is unavailable.
- **Chat mode**: Claude gets a `semantic_search` tool that calls the same function. Replaces the current FTS-based `search_notes` tool.

### Cost

~$0.0001 per embedding. A heavy day of 100 note saves = ~$0.01.

### Existing Notes

No migration needed — the app starts fresh with no existing notes. Every note gets embedded as it's created going forward.

## 3. Chat System

### Mode Switching

- `/chatmode` — switches to chat. Opens most recent conversation, or creates one if none exists.
- `/notemode` — switches back to the editor.
- `/newchat` — starts a fresh conversation (available in chat mode).
- App tracks `mode: "notes" | "chat"` in page state. Starts in notes mode.

### Chat Layout

Messages displayed chronologically in the same editor frame. Same visual weight as notes mode with a small differentiator (TBD in implementation — could be a subtle background tint, mode indicator, or message styling). It should feel like part of the same app, not a chatbot widget.

### Conversation Lifecycle

- `/chatmode` opens the most recent conversation, or creates new
- `/newchat` starts a fresh conversation
- Old conversations are kept but no conversation browser for now — just current and new
- Conversation title auto-generated from first user message

### Chat API Route

New route: `/api/ai/chat/route.ts`

1. Receive user message + conversationId
2. Load last ~20 messages from conversation for context (token window management)
3. System prompt includes: vault tools, semantic search, person system instructions
4. Tool-use loop (same pattern as existing Ask Claude and `/claude` commands)
5. Stream assistant response back
6. Save both user and assistant messages to Message table

### Tools Available in Chat

| Tool | New/Existing | Description |
|------|-------------|-------------|
| `semantic_search(query)` | New | Embedding-based search |
| `read_note(id)` | Existing | Read full note content |
| `create_note(title, content)` | Existing | Create a new note |
| `update_note(id, ...)` | Existing | Update existing note |
| `list_people()` | Existing | List all people with mention counts |
| `update_person(name, content)` | New | Update a person's note (for observations like "Sarah is worried about Q2") |
| `create_pending_person(name, context)` | New | Flag unrecognized name for user review |

## 4. People System Overhaul

### `/people` Command (Enhanced)

Opens people modal sorted by most recent interaction (last note or chat linking). Each entry shows: name, role, note count. Click opens the person page.

### Person Page

Two sections:

**Top: AI-maintained summary**
- Stored in `PersonMeta.summary`
- Regenerated fire-and-forget whenever a new note or chat message links to this person
- New route: `/api/ai/person-summary/route.ts`
- Input: person name, role, userContext, all connected notes (content truncated to ~500 chars each)
- Output: narrative summary — relationship context, what they care about, the user's dynamic with them
- User can edit the summary directly. The summary field is always the source of truth — when AI regenerates, it receives the current summary as input and produces an updated version that incorporates new information while preserving the existing tone and content

**Bottom: Connected notes stream**
- Chronological list of every note linked via NotePerson
- Each entry: note title, date, AI-extracted highlight (most relevant sentence about this person)
- Highlights stored on `NotePerson.highlight`, generated at link-time (when organize or chat links a note to a person)

### `/newperson` Command (Conversational Inline Flow)

1. `/newperson` triggers a stepped prompt in the editor area
2. **Name?** — user types, Enter to continue
3. **Role?** — user types, Enter to continue
4. **Anything else about them?** — user types freely, or Enter to skip
5. Creates Note (type: "person") + PersonMeta with name, role, userContext
6. If userContext provided, AI generates initial summary from it (fire-and-forget)
7. Aliases auto-generated: full name + first name (e.g., "Sarah Chen" produces aliases ["Sarah Chen", "Sarah"])

### `/pendingpeople` Command

Opens modal listing PendingPerson entries with status "pending". Each entry shows:
- Name
- Source context (e.g., "mentioned in *Meeting Notes March 3*: 'John said the timeline is tight'")

Actions per entry:
- **Confirm** — launches `/newperson` flow pre-filled with the name
- **Merge** — shows people list, user picks existing person to add name as alias
- **Dismiss** — marks as dismissed, won't resurface

### AI Detection of New Names

- In organize: unrecognized names create PendingPerson entries (not person notes)
- In chat: `create_pending_person` tool instead of silent person creation
- Known names (matched by alias) still auto-link via NotePerson as before

## 5. AI Organize Changes

The organize system gets simpler.

### Removed from Organize Output

- `tags` array — AI no longer appends tags to note content
- `tagCorrections` — no more AI-driven tag spelling fixes

### Kept

- `links` — still appends `[[wiki-links]]` to connect related notes
- People detection — links recognized names via NotePerson, creates PendingPerson for unrecognized names

### New Side Effects

- After organize updates a note, the embedding pipeline fires on the save
- If organize links a new person to the note, that person's summary gets regenerated (fire-and-forget)

### System Prompt Changes

- Remove tag vocabulary from context
- Remove tag-related instructions
- Keep: people list, recent note titles, sibling context
- Add instruction: "If you encounter a name that doesn't match any known person, create a pending person entry — do not create a person note."

### Trigger Unchanged

Fire-and-forget on note close (2s debounced) and via `/organize` slash command.

## 6. Technical Decisions

### Embedding Model

Voyage AI `voyage-3` — 1024 dimensions, highest retrieval quality benchmarks, recommended by Anthropic.

### Vector Storage

Binary blob in SQLite Embedding table. Float32Array serialized. No vector database — brute-force cosine is exact (no approximation loss) and fast at personal knowledge base scale.

### Chat AI Model

Same as existing: `claude-sonnet-4-20250514` via Anthropic API. Tool-use loop pattern reused from existing routes.

### Embedding on Save vs. Background Queue

Embedding happens fire-and-forget on every save. No job queue — if it fails, it retries on the next save. Simplicity over reliability for a personal tool.

### Person Summary Regeneration

Fire-and-forget on new person-note link. Same conditional update pattern as organize (snapshot updatedAt, discard if changed). No queue.
